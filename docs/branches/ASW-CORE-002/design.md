# Phase V0.2: Corporate Memory & Human Resources

## 1. Objective

Formalize the `.company/` file-system database, enforce organizational standards via prompt injection, and implement the two-step Hiring pipeline (roster planning + per-role generation). This phase builds directly on V0.1's CPO → CTO pipeline.

### V0.1 Recap

The V0.1 pipeline (`asw start`) runs two phases:

- **Phase A (CPO):** Vision → PRD (`prd.md`) with Founder Gate
- **Phase B (CTO):** Vision + PRD → Architecture (`architecture.json` + `architecture.md`) with Founder Gate

V0.2 adds **Phase C** (two sub-phases) after Phase B, expands the `.company/` directory, and introduces standards injection.

---

## 2. The Directory Structure Database

The `.company/` directory is the agents' shared reality — a vector-less memory store on the file system.

### V0.2 Layout

```text
.company/
├── roles/          ← Markdown system prompts (bundled + generated)
├── artifacts/      ← Pipeline outputs (prd.md, architecture.json, roster.json, etc.)
├── memory/         ← Living documents generated by the workflow (renamed from state/)
├── templates/      ← Reusable .md structures copied from bundled defaults
└── standards/      ← Organizational rules injected into agent system prompts
```

### Changes from V0.1

| Directory | V0.1 | V0.2 |
|-----------|------|------|
| `roles/` | `cpo.md`, `cto.md` (Markdown) | + `hiring_manager.md`, `role_writer.md`, and dynamically generated role files (all Markdown) |
| `artifacts/` | `prd.md`, `architecture.json`, `architecture.md` | + `roster.json`, `roster.md` |
| `state/` | Created but unused | **Renamed to `memory/`** — migration handled automatically by `init_company()` |
| `templates/` | Did not exist | New: `prd_template.md`, `architecture_template.md`, `role_template.md` |
| `standards/` | Did not exist | New: `python_guidelines.md`, `ui_guidelines.md` |

### Migration

`init_company()` detects an existing `.company/state/` directory and renames it to `.company/memory/` automatically. No manual intervention required.

## 3. Standards Injection

Standards files in `.company/standards/` contain organizational rules that are appended to agent system prompts.

### Mechanism

The `Agent` dataclass gains an optional `standards: list[Path]` field (default: empty list). When `Agent.run()` is called:

1. The role file is read as the base system prompt.
2. Each standards file path in `self.standards` is read and appended to the system prompt, separated by `\n\n---\n\n`.
3. The combined prompt is sent to the LLM as the system message.

```python
@dataclass
class Agent:
    name: str
    role_file: Path
    llm: LLMBackend
    standards: list[Path] = field(default_factory=list)
```

### Bundled Standards

| File | Contents |
|------|----------|
| `python_guidelines.md` | Google docstrings, comprehensive unit tests, standard libraries, type hints, error handling conventions |
| `ui_guidelines.md` | Dark theme default, accessible contrast ratios, responsive design principles |

Standards are copied from the bundled package into `.company/standards/` on `init_company()` (skip if already present, same as roles). The Founder can edit them between runs.

## 5. The Hiring Pipeline (Phase C)

Phase C is split into two sub-phases:

### Phase C1: Roster Planning (Founder Gate)

The Hiring Manager agent reads the CTO's `architecture.json` and proposes a roster of specialist roles needed to implement the system.

**Input context:**

- `architecture`: contents of `architecture.json`
- `available_standards`: list of filenames in `.company/standards/`

**Output:** A single fenced ` ```json ``` ` block containing:

```json
{
  "hired_agents": [
    {
      "title": "Security Architect",
      "filename": "security_architect.md",
      "responsibility": "Enforce OWASP Top 10 and design secure authentication flows",
      "assigned_standards": ["python_guidelines.md"]
    },
    {
      "title": "Python Backend Developer",
      "filename": "python_backend_developer.md",
      "responsibility": "Implement API endpoints and business logic",
      "assigned_standards": ["python_guidelines.md"]
    },
    {
      "title": "Frontend Developer",
      "filename": "frontend_developer.md",
      "responsibility": "Build responsive UI components",
      "assigned_standards": ["ui_guidelines.md"]
    }
  ]
}
```

**Mechanical linting (`_lint_roster`):**

- Fenced JSON block must be present and parseable
- `hired_agents` must be a non-empty array
- Each entry must have: `title` (str), `filename` (str), `responsibility` (str), `assigned_standards` (list of str)
- Filenames must match `^[a-z][a-z0-9_]*\.md$`
- Each `assigned_standards` entry must reference a file that exists in `.company/standards/`

**Rendering:** `_render_roster_markdown(json_str)` converts the roster JSON into a human-readable Markdown table (same pattern as `_render_architecture_markdown`):

```markdown
# Proposed Roster

> **Source of Truth:** The roster specification is stored in `roster.json`.

| # | Title | Filename | Responsibility | Standards |
|---|-------|----------|----------------|-----------|
| 1 | Security Architect | security_architect.md | Enforce OWASP Top 10... | python_guidelines.md |
| 2 | Python Backend Developer | python_backend_developer.md | Implement API endpoints... | python_guidelines.md |
| 3 | Frontend Developer | frontend_developer.md | Build responsive UI... | ui_guidelines.md |

**Total: 3 roles proposed**
```

**Founder Gate:** The Founder reviews `roster.md` and can:

- **Approve:** Proceed to role generation
- **Reject:** Re-run the Hiring Manager from scratch
- **Modify:** Directly edit the roster JSON (add/remove/rename roles). The edited JSON is re-validated via `_lint_roster` before saving.
- **Stop:** Exit the pipeline

**Artifacts written:** `roster.json` and `roster.md` to `.company/artifacts/`

### Phase C2: Role Generation (Automatic)

After the roster is approved, the pipeline iterates through each role and generates a Markdown system prompt via the **Role Writer** agent — one LLM call per role, no Founder Gate.

**Role Writer input context (per role):**

- `architecture`: contents of `architecture.json`
- `role_title`: the role's title from the roster
- `role_responsibility`: the role's responsibility from the roster
- `role_template`: contents of `.company/templates/role_template.md`
- `assigned_standards`: contents of each assigned standards file

**Role Writer output:** A complete Markdown system prompt for the specified role, following the structure defined in `role_template.md`.

**Mechanical linting (`_lint_role`):**

- Must start with `# Role:` heading
- Must contain `## Output Format` section
- Must contain `## Strict Rules` section
- Must be non-empty (minimum 200 characters)

**For each role in the roster:**

1. Create Role Writer agent with role metadata as context
2. Run through `_agent_loop` with `_lint_role` (up to 3 attempts)
3. Write generated `.md` file to `.company/roles/<filename>`
4. Print confirmation: `✓ Generated role: <title> → <filename>`

**After all roles:** Print summary: `✓ Generated N role files`

### Pipeline Integration

```text
run_pipeline()
├── Phase A: CPO → PRD (unchanged)
├── Phase B: CTO → Architecture (returns architecture JSON string)
├── Phase C1: Hiring Manager → Roster
│   ├── _agent_loop(Hiring Manager, {architecture, available_standards}, _lint_roster)
│   ├── Write roster.json + roster.md
│   ├── founder_review("Roster", roster.md) → approve/reject/modify loop
│   └── Return approved roster
├── Phase C2: Role Generation
│   ├── For each role in roster:
│   │   ├── _agent_loop(Role Writer, {architecture, role_metadata, template, standards}, _lint_role)
│   │   └── Write .company/roles/<filename>.md
│   └── Print summary
└── git commit "hiring completed"
```

The `_run_architecture_phase` function is refactored to return the architecture JSON string so it can be passed to Phase C1.

## 5b. Pipeline Resume & Restart

By default, the pipeline resumes from the point at which it last stopped. This saves LLM tokens by avoiding regeneration of already-approved artifacts.

### State Tracking

Pipeline progress is tracked via `.company/pipeline_state.json`:

```json
{
  "version": "0.2",
  "vision_sha256": "a1b2c3…",
  "completed_phases": {
    "prd": {"timestamp": "2026-04-18T12:00:00+00:00"},
    "architecture": {"timestamp": "2026-04-18T12:05:00+00:00"},
    "roster": {"timestamp": "2026-04-18T12:10:00+00:00"},
    "roles": {"timestamp": "2026-04-18T12:15:00+00:00"}
  }
}
```

State is written **after** each phase completes (post-founder-gate, post-artifact-write). A crash mid-phase does not mark it complete.

### Resume Logic

On each `run_pipeline()` invocation:

1. Read existing `pipeline_state.json` (if present).
2. Hash the vision file (SHA-256) and compare with stored `vision_sha256`.
3. If the vision changed, prompt the user: **[C]ontinue** from where you left off, or **[R]estart** from scratch.
4. For each phase, check two conditions:
   - Phase is recorded as completed in state.
   - All expected artifact files exist on disk.
5. If both conditions are met, **skip** the phase and load existing artifacts from disk for downstream context.
6. If either condition fails, **run** the phase normally.

```text
run_pipeline() — with resume
│
├── Read pipeline_state.json
├── Hash vision file → compare with stored hash
├── If changed → prompt user (Continue / Restart)
│
├── Phase A: _is_phase_done("prd", [prd.md])?
│   ├── Yes → load prd.md from disk, print "↩ Skipping PRD phase"
│   └── No  → run CPO agent, write prd.md, mark_phase_complete("prd")
│
├── Phase B: _is_phase_done("architecture", [architecture.json, architecture.md])?
│   ├── Yes → load architecture.json from disk, print "↩ Skipping Architecture phase"
│   └── No  → run CTO agent, write artifacts, mark_phase_complete("architecture")
│
├── Phase C1: _is_phase_done("roster", [roster.json, roster.md])?
│   ├── Yes → load roster.json from disk, print "↩ Skipping Roster phase"
│   └── No  → run Hiring Manager, write artifacts, mark_phase_complete("roster")
│
├── Phase C2: _is_phase_done("roles", [])?
│   ├── Yes → print "↩ Skipping Role Generation phase"
│   └── No  → run Role Writer per role, mark_phase_complete("roles")
│
└── Done
```

### Restart

The `--restart` CLI flag deletes the entire `.company/` directory before the pipeline begins:

```bash
asw start --vision vision.md --restart
```

This gives a clean slate — all generated artifacts, customised bundled assets, and state are removed. `init_company()` then recreates the directory structure and re-copies bundled defaults.

### State Functions (in `company.py`)

| Function | Purpose |
|----------|---------|
| `hash_file(path)` | SHA-256 hex digest of a file |
| `read_pipeline_state(workdir)` | Read and parse `.company/pipeline_state.json`; returns `None` if missing or corrupt |
| `write_pipeline_state(workdir, state)` | Write state dict to `.company/pipeline_state.json` |
| `mark_phase_complete(workdir, state, phase)` | Record phase completion with timestamp and persist |
| `clear_company(workdir)` | Delete entire `.company/` directory |

## 6. Bundled Role Files

Two new role files are bundled with the package:

### `hiring_manager.md`

System prompt for the Hiring Manager agent. Key instructions:

- Analyse the architecture JSON to determine what specialist roles are needed
- Assign relevant standards from the available list
- Use `lowercase_underscore.md` filenames
- Output exactly one fenced JSON block with the roster schema
- Do not invent roles beyond what the architecture requires

### `role_writer.md`

System prompt for the Role Writer agent. Key instructions:

- Generate a complete Markdown system prompt for one specific role
- Follow the structure in `role_template.md`
- Be specific to the role's responsibility and the project's architecture
- Include clear output format requirements and strict rules
- Output only the role file content — no preamble or sign-off
